## 1. 系统架构与职责边界（实现事实）

### 1.1 后端形态：Python FastAPI 为主，旧 Node/Express 保留

- Python FastAPI 入口为 [mining-plan/backend_python/main.py](mining-plan/backend_python/main.py#L1-L78)，在同一进程中注册多个业务路由：upload/boreholes/design/score/boundary/geology/succession/gnn/planning/export（实现锚点：[mining-plan/backend_python/main.py](mining-plan/backend_python/main.py#L11-L67)）。
- legacy Node/Express 入口为 [mining-plan/backend/index.js](mining-plan/backend/index.js#L1-L30)，注册 boundary/boreholes/score/design/upload/geology 等路由，但未包含 smart-planning 路由（实现锚点：[mining-plan/backend/index.js](mining-plan/backend/index.js#L1-L24)）。

### 1.2 前端形态：Vite dev server +（可选）后端同端口静态托管

- 前端 dev server 启动脚本为 [mining-plan/tools/start-frontend.ps1](mining-plan/tools/start-frontend.ps1#L1-L23)，以 `npm run dev -- --host 0.0.0.0 --port 5173 --strictPort` 方式启动。
- 后端在检测到 `frontend/dist` 存在时，会把静态站点 mount 到 `/`（用于 3001 同端口演示），且要求放在 `/api` 路由之后避免吞掉 API（实现锚点：[mining-plan/backend_python/main.py](mining-plan/backend_python/main.py#L92-L112)）。

## 2. 问题形式化（面向论文表述）

> 只给“可从实现推导”的最小形式化；更细的目标/约束分层与参数来源见智能规划专用文档。

### 2.1 设计对象（输出）

- 设计输出以“工作面条带集合（loops）”为核心表示：
  - smart-efficiency/smart-resource 在 worker 输出 `candidates[]`，每个 candidate 携带 `render.plannedWorkfaceLoopsWorld`（或裁剪后 `render.clippedFacesLoops`）等几何（证据链总览见智能规划专用文档的 2/6 节）。

### 2.2 目标函数与多目标组合（weighted 事实口径）

- smart-weighted 后端输入权重 `weights(efficiency,recovery,disturbance)`，并提供 `disturbanceParams` 的默认值（样本步长/阈值/分位数权重等）（实现锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L210-L259)）。
- smart-weighted 的实现：候选池拼接 + recovery 评分（tonnage endpoint）+ disturbance 评分（ODI 场采样）+ NSGA-II TopK 选择 + 加权综合得分（实现锚点总入口：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L1269-L1808)；细节锚点见智能规划专用文档第 5 节 C1~C4）。

## 3. 约束分层（工程实现口径）

- 输入合法性与快速失败：
  - smart-efficiency 对边界点数与工作面宽度范围做硬校验并写入 `attemptSummary.failTypes`（实现锚点见智能规划专用文档第 4.1 节）。
  - smart-resource 对 `tonnageTotal` 顶层字段做“回包字段契约”（前端硬校验需要该字段存在）（实现锚点见智能规划专用文档第 6.2 节）。
- 几何可行性：条带求交、可行区间求解、矩形构造与 failTypes 统计（实现锚点见智能规划专用文档第 4.2 节）。
- 工程验收口径：
  - smart-efficiency：`COVERAGE_MIN=0.70` 用于 qualified 与兜底 warning（实现锚点见智能规划专用文档第 4.3/ A7）。
  - smart-resource：`fullCoverMin` 默认 0.995 + `ignoreCoalPillarsInCoverage`，并提供 cleanupResidual 与 fullCoverPatch 的工程增强（实现锚点见智能规划专用文档第 4.4/ B7~B8）。

## 4. 方法框架（端到端证据链）

### 4.1 UI → API → FastAPI → Node harness → Worker（智能规划）

- 路由定义：FastAPI planning router 暴露 `/planning/smart-efficiency/compute`、`/planning/smart-resource/compute`、`/planning/smart-weighted/compute`、`/planning/smart-resource/tonnage`、`/planning/l2/compare*`（实现锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L1114-L1216)，[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L1269-L1276)，[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L2513-L2516)）。
- 统一前缀：planning router 在 FastAPI 中以 `prefix="/api"` 注册，因此最终 HTTP 路径为 `/api/planning/...`（实现锚点：[mining-plan/backend_python/main.py](mining-plan/backend_python/main.py#L58-L67)）。
- Node harness 复用 worker：后端通过子进程运行 `tools/run-smart-*.mjs` 来执行前端 worker 代码并回传结果（实现与稳定窗口锚点见智能规划专用文档第 2/3 节）。

### 4.2 legacy Node/Express 设计流（非 smart-planning）

- legacy `/api/design` 走 Node 后端的“工作面布局算法”链路：路由在 [mining-plan/backend/routes/design.js](mining-plan/backend/routes/design.js#L1-L140)；核心布局函数调用 `layoutWorkfaces(...)`（调用锚点：[mining-plan/backend/routes/design.js](mining-plan/backend/routes/design.js#L69-L92)）。
- `layoutWorkfaces` 的实现位于 [mining-plan/backend/utils/workfaceDesign.js](mining-plan/backend/utils/workfaceDesign.js#L368-L616)，并依赖一组规程口径的几何/参数计算（例如 `selectStartBoundary/calculatePillarWidth/calculateWorkfaceWidth/calculateWorkfaceLength` 等）。

## 5. 输入/输出 Schema 与默认值（可复现“参数来源”）

### 5.1 后端 planning（Pydantic 模型）

- smart-weighted 请求模型 `SmartWeightedComputeRequest`：含 `effCacheKey/recCacheKey/distCacheKey`、`weights`、`odiFieldPack`、`disturbanceParams` 等（实现锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L226-L259)）。
- disturbanceParams 默认值（后端声明口径）：`sampleStepM=25,maxSamples=4500,exceedThreshold=0.7,wMean=0.50,wP90=0.35,wExceed=0.15,outerBufferM=30`（实现锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L216-L224)）。
- tonnage 排序请求 `SmartResourceTonnageRequest`：支持 `preferPatched/cleanupResidual`（用于跨模式吨位可比）与 `wTonnage/wCoverage/wEngineering/fullCoverMin` 等评分旋钮（实现锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L161-L187)）。

### 5.2 前端 planning 请求参数来源（UI）

- smart-efficiency：UI 默认 `searchProfile:'balanced'`，并在 disturbance 下追加 `maxTimeMs:180000,maxCandidates:400` 等性能旋钮（调用锚点：[mining-plan/frontend/src/App.jsx](mining-plan/frontend/src/App.jsx#L4157-L4189)）。
- smart-resource：UI 显式传入 `maxTimeMs:30000`、`fullCoverPatchMaxTimeMs:3000`、`fullCoverMin:0.995`、以及 `segmentWidth/cleanupResidual` 的工程参数（调用锚点：[mining-plan/frontend/src/App.jsx](mining-plan/frontend/src/App.jsx#L4700-L4795)）。

## 9. Node vs Python 实现对比（仓库事实）

- Python FastAPI（planning）并不直接重写 smart-efficiency/smart-resource 的核心几何搜索：它通过 Node harness 运行前端 worker 以保持结果一致（证据链详见智能规划专用文档第 2/3 节，后端路由锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L1156-L1276)）。
- legacy Node/Express 提供的是另一条“设计/评分/地质处理”的 API 形态，与 smart-planning 不同（入口与路由锚点：[mining-plan/backend/index.js](mining-plan/backend/index.js#L1-L24)，设计路由锚点：[mining-plan/backend/routes/design.js](mining-plan/backend/routes/design.js#L1-L92)）。

## 10. 局限性（仅基于实现可见事实）

- 多处缓存为进程内内存缓存（Node 结果缓存与 weighted 缓存），重启后不保留，且上限固定（实现锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L17-L74)，[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L84-L118)）。
- smart-resource 的工程增强（cleanupResidual/fullCoverPatch）会改变 top1 与可视化 loop，且需要在 UI/后端评分联动时明确“preferPatched”口径，否则可能出现“同 candidate 不同显示形状导致指标不一致”的表象（tonnage 请求支持 `preferPatched/cleanupResidual` 的事实锚点：[mining-plan/backend_python/routers/planning.py](mining-plan/backend_python/routers/planning.py#L161-L187)）。
