# 智能规划（四模型）算法逻辑说明

> 面向：开发/验收/维护
>
> 目标：把当前“智能规划”板块四种模式（工程效率 / 覆岩扰动 / 资源回收 / 权重综合）的**候选生成、约束口径、评分与排序、缓存与一致性策略**落成一份可复用说明。
>
> 事实来源：以代码实现为准（见“实现位置索引”）。本文只做“口径归纳”，不改变现有算法。

---

## 1. 总览：四模式的关系与职责边界

### 1.1 四种模式做的事情

- **工程效率（efficiency）**：在采区边界内生成可行工作面布置候选，并以“覆盖率绝对优先 + 次级工程效率评分”排序。
- **覆岩扰动（disturbance）**：给候选布置计算“ODI 扰动暴露风险”，并按风险由低到高排序。
  - 目前有两条实现路径：
    - **前端派生**：对“工程效率”候选做二次评价（更贴近 UI 展示）；
    - **worker 搜索（disturbance mode）**：用序列枚举生成更“扰动多样性”的候选池，供 weighted 用于扩充候选。
- **资源回收（recovery）**：在边界内生成候选布置，并结合厚度场估算吨位，形成回收评分与排序。
- **权重综合（weighted）**：后端将 efficiency/recovery/(可选)disturbance 三套候选合并成候选池，计算三目标（效率/回收/扰动）后用 NSGA-II 选 TopK，再按用户权重合成分排序。

### 1.2 前后端分工（口径一致性）

- 前端：
  - UI 状态/缓存键口径固化、触发计算、展示与交互。
  - disturbance 在 UI 上默认走“基于 efficiency 的派生评分”（更快、展示一致）。
- 后端（FastAPI）：
  - **smart-efficiency / smart-resource**：通过 Node harness 复用前端 worker 算法，保证口径一致。
  - **smart-weighted**：后端独立实现多目标融合与选择（NSGA-II + 权重合成），并调用后端吨位/回收评分辅助。

## 3. 共性输入与核心术语

### 3.1 几何与布置参数（核心）

- `boundaryLoopWorld`：采区边界闭合环（world 坐标）。
- `axis`：布置方向（`x` 或 `y`）。注意：厚度坐标系不随 axis 改变（见 5.2）。
- `wb`（boundary pillar）：边界煤柱宽度。
- `ws`（section pillar）：区段煤柱宽度。
- `B`：工作面宽度。
- `N`：工作面数量。

### 3.2 候选对象通用字段（典型）

不同模式/来源候选字段略有差异，但 weighted 会尽量从以下字段提取：

- `signature`：候选唯一签名（排序/缓存/去重关键）。
- `qualified`：是否满足硬约束（硬约束不满足也可能保留用于 UI 观察/回退展示）。
- `coverageRatio` / `coverageRatioEff`：覆盖率（raw/eff 口径见 4.4、6.2）。
- `efficiencyScore`：工程效率评分（v2）。
- `tonnageTotal`：估算吨位（来自厚度场或回退）。
- `recoveryScore`：回收综合得分（0-100）。
- `render.*`：几何渲染用 loops（faces/union/omega 等）。

## 4. 模式一：工程效率（efficiency）

### 4.1 总流程

1) 预处理：边界点清洗/闭合/修复（buffer(0) 等）、构造采区 polygon。
2) 生成内缩 Ω（有效范围）：按边界煤柱 `wb` 对边界内缩，得到可布置区域。
3) 参数采样与搜索：对 `ws`、`N`、`B` 组合进行搜索与评估（见 4.2）。
4) 对每个可行组合生成候选，计算覆盖率与工程效率评分。
5) 排序：覆盖率绝对优先（epsilon 分档），再按次级指标 tie-break（见 4.3）。
6) 输出：候选列表、Top1、以及用于 UI 快速切换 N 的索引（bestKeyByN / candidatesByN）。

### 4.2 搜索口径（关键点）

- cacheKey 口径在前端固化（`buildEfficiencyCacheKey`）：
  - **wb 固定取最小值**：`wbRep = min(wbMin, wbMax)`
  - **ws 采用 coarse9 代表点采样**（“均衡档近似”）：
    - 端点 + target + 等分插值，落到整数米并去重
  - **B 搜索为“粗到细”两阶段**：先粗步长（例如 10m），再局部 1m 精修（win/seed 等参数）
  - Ncap（默认 20）限制组合数量

> 目的：在保证口径稳定的前提下，减少重几何评估次数，提升可交互性。

### 4.3 评分与排序

#### 4.3.1 工程效率评分 v2（用于 tie-break/展示）

在 `smartEfficiency.worker.js` 中：

- 基础分：`base = coverageRatio * 100`
- 惩罚项：
  - `penaltyN = wN * (N-1)`
  - `penaltyCV = wCV * clamp(lenCV, 0, 2)`（推进长度离散惩罚）
  - `penaltyShort = wShort * shortRatio`，其中 `shortRatio = clamp((minLRef - minL)/minLRef, 0, 1)`
- 最终：`efficiencyScore = base - (penaltyN + penaltyCV + penaltyShort)`

权重（固化在 cacheKey 中以隔离版本）：
- `wN=0.20, wCV=10.0, wShort=5.0, minLRef=100`

#### 4.3.2 排序规则：覆盖率绝对优先（coverageFirst）

核心思想：
- 覆盖率差异小于 `COVERAGE_EPS=1e-5` 视为同档，进入 tie-break。
- tie-break 顺序（概念层）：
  1) 覆盖率档位（大优先）
  2) N（小优先）
  3) efficiencyScoreV2（大优先）
  4) 其它稳定性指标（例如 lenCV/minL/B 等）

> 目标：避免几何浮动导致排序抖动，同时把“覆盖率最大化”作为硬性第一优先级。

### 4.4 覆盖率口径（raw vs eff）

前端 UI 有“统一展示口径”的处理：
- 在 recovery 模式下默认展示 raw 覆盖率（避免分母差异造成“看起来回收模式覆盖率更大”）。
- fullCover/残煤判断仍基于 eff 覆盖率。

## 5. 模式二：覆岩扰动（disturbance）

### 5.1 UI 派生扰动（基于 efficiency 结果）

入口：`frontend/src/App.jsx` 的 `buildDisturbanceRankingPackForEfficiencyResult(effResult, odiPack, params)`。

#### 5.1.1 采样区域构造

对每个候选：
- 优先使用“实际采出区域”loops（按优先级）：
  1) `render.clippedFacesLoops`
  2) `render.plannedWorkfaceLoopsWorld`
  3) `render.facesLoops`
  4) 回退：`render.rectLoops`（规则矩形近似）
- 与有效 Ω 相交裁剪（若候选携带 `omegaEffLoopsWorld` 则优先用之，否则回退到结果级 Ω）。
- 支持 “采出区域外扩” 参数 `outerBufferM`（默认 30m）：
  - 先对 mined 区域 buffer，再**裁剪回有效 Ω**（体现超采出影响，但不跑出有效范围）。

#### 5.1.2 ODI 采样与自适应步长

- 基础采样步长：`sampleStepM`（默认 25m）。
- 上限：`maxSamples`（默认 4500）。
- 若 bbox 粗估采样点过多，会自动增大步长（目标：避免卡死）。

#### 5.1.3 扰动统计量与 rawScore

对采样得到的 ODI 值集合 $V$：
- 均值：$\mu = mean(V)$
- 90 分位：$p90 = quantile(V, 0.90)$
- 超阈比例：$exc = |\{v\in V: v\ge thr\}|/|V|$，默认 `thr=0.7`

加权 rawScore（**越小越好**）：

$$
score = w_{mean}\cdot \mu + w_{p90}\cdot p90 + w_{exc}\cdot exc
$$

默认权重：
- `wMean=0.50, wP90=0.35, wExceed=0.15`

#### 5.1.4 distPoints（展示分）标定

- 用所有候选 rawScore 的 5%/95% 分位做标定区间：`lo=q05, hi=q95`
- 线性映射到 `[best=95, worst=60]`，再 clamp 到 `[0,100]`

> 注意：distPoints 的“绝对值”不代表工程意义的绝对阈值，它是**相对标定**，主要服务 UI 观感与跨候选对比。

#### 5.1.5 排序（UI 扰动表）

默认策略：
1) `qualified` 优先
2) `rawScore` 升序（低扰动优先）
3) `coverage` 降序
4) 其它稳定性 tie-break（保持排序稳定）

### 5.2 worker 扰动搜索（optMode=disturbance）

入口：`frontend/src/planning/workers/smartEfficiency.worker.js` 的 `optMode === 'disturbance'` 分支。

关键口径（由 `buildDisturbanceCacheKey` 固化）：
- **wb 固定取最大值**：`wbRep = max(wbMin, wbMax)`
- **ws 1m 全枚举**（范围内整数米逐一尝试）
- 允许 `BSeq/WsSeq` 序列枚举：
  - `BSeq` 长度为 N
  - `WsSeq` 长度为 N-1
  - 邻接约束：`BAdjMax=30`（相邻面宽变化不超过 30）
  - 候选数量上限：`maxCand=400`

> 这条路径主要用于“增加候选多样性/扰动多样性”，并不等同于 UI 上展示的“扰动评分表”。

## 6. 模式三：资源回收（recovery）

### 6.1 总流程

1) 候选生成：`smartResource.worker.js` 在 Ω 内生成布置候选（含 per-face 变形、分段变宽、残煤清扫等工程化约束）。
2) 吨位估算：
   - 前端可调用后端 `/planning/smart-resource/tonnage` 对候选批量评估（推荐，口径集中、可缓存）。
3) 回收评分：综合吨位、覆盖率、工程性，得到 `recoveryScore (0..100)`。
4) 排序：`qualified` 优先，然后 recoveryScore、吨位、覆盖率、N 等。

### 6.2 厚度场协议（v1.0 冻结）

在 `smartResource.worker.js` 中冻结：

- `fieldPack.field[j][i]`：二维数组
  - `i ∈ [0, gridW-1]`
  - `j ∈ [0, gridH-1]`
- 行方向：`j=0` 对应 `y≈ymax`（上边界），j 增大表示 y 从 ymax → ymin（自上向下）
- 采样：双线性插值 + y 翻转映射
- **axis 只影响工作面布置，不影响厚度坐标系**（axis=y 不会对厚度场 swapXY）

采样器优先级：
1) grid（field 有效且存在 >0 的有效 cell）
2) constantM（应急兜底）
3) none（无厚度信号）

### 6.3 后端吨位与回收评分（smart_resource_tonnage）

入口：`backend_python/routers/planning.py` 的 `@router.post('/planning/smart-resource/tonnage')`。

#### 6.3.1 吨位估算

- 默认采用网格采样（`sampleStepM` 默认优先 `thickness.gridRes`，前端常传 20）。
- 只在 mined 区域内取样：`tonnage = Σ(thk(x,y) * step^2 * rho)`。

#### 6.3.2 工程性子分（0..1）

后端从候选提供的稳定信号构造工程性：
- 推进离散 `lenCV`（越小越好）
- 异常面数 `abnormalFaceCount`（越小越好）
- 面宽波动 `BMax-BMin`（越小越好）
- 面数 N（4..7 视作更优区间）

综合：
- `s_eng = 0.35*s_len + 0.25*s_abn + 0.20*s_b + 0.20*s_n`

#### 6.3.3 recoveryScore（0..100，展示校准）

默认权重：
- `wTonnage=0.55, wCoverage=0.30, wEngineering=0.15`

细节：
- 吨位归一化：对 `tonnage` 做 `log1p` 再 min-max（降低极端值支配）。
- 若没有厚度信号（fallback AREA），会把权重在 coverage+engineering 内重归一化。
- `fullCoverMin` 惩罚：仅对 coverage 子项做温和惩罚（smoothstep + 最低系数 0.85），不再把总分整体系数乘小。
- 展示校准：
  - `base∈[0,1] -> calibrated = 0.45 + 0.55*base`
  - `recoveryScore = 100*calibrated`（再 clamp 到 <=98）

> 这一步是“产品化观感”处理：保留相对差异，同时让 Top1 更自然落在 90+ 区间。

## 7. 模式四：权重综合（weighted，多目标融合）

入口：`backend_python/routers/planning.py` 的 `smart_weighted_compute`（`/planning/smart-weighted/compute`）。

### 7.1 核心策略

1) **候选池生成（复用 worker 口径）**
   - 用 Node harness 并行运行：
     - smart-efficiency（input.mode=efficiency）
     - smart-resource（input.mode=recovery）
     - 可选：smart-efficiency（input.mode=disturbance，用于多样性，只有在 hasODI 且 wDist>0 时）
2) **合并候选池**
   - 优先取 `qualified=true` 的候选；如果一个都没有，则回退展示不合格候选（用于 UI 排查）。
3) **对候选做三目标评估**
   - recovery：调用 `smart_resource_tonnage` 批量计算 `tonnageTotal` 与 `recoveryScore`。
   - disturbance：若有 ODI 场，调用 `_compute_disturbance_for_candidates` 得到 `distPoints`（0..100，越大越好）。
   - efficiency：直接使用候选携带的 `efficiencyScore`。
4) **NSGA-II 选择 TopK**
   - 非支配排序（maximize）+ crowding distance。
5) **权重合成分排序**
   - 在“已选 TopK 集合”内做 min-max 归一化（提高稳定性）。
   - `totalScore = wEff*effNorm + wRec*recNorm + wDist*distNorm`。

### 7.2 权重处理与“严格等价”

- 如果**没有 ODI 场**：强制 `wDist=0`，并对 `wEff/wRec` 二维归一化。
- 如果用户把权重设为“单目标”（例如 efficiency=1, 其余=0）：
  - 后端进入 strict 模式：只运行对应模式的 node harness，并且**不对 cacheKey 做 salt**。
  - 目的：做到“weighted 单目标结果”与对应单模式结果严格一致（候选池、随机种子等对齐）。
- 前端会把 `effCacheKey/recCacheKey/distCacheKey` 一并传入 weighted，用于后端对齐种子。

### 7.3 扰动评分（weighted 侧）

weighted 侧的 `_compute_disturbance_for_candidates` 与 UI 派生的核心统计一致：
- polygon 内 grid sampling
- `score = wMean*mean + wP90*p90 + wExceed*exceedRatio`
- 用 q05/q95 把 score 映射为 `distPoints∈[60,95]`（再 clamp 0..100）

差异点：
- weighted 的采样区域来自候选的 `render` loops（优先 faces/union/omega 等可用字段）。
- weighted 返回 distPoints 用于优化（越大越好），并额外带 `distScore/distMean/distP90/distExceedPct` 用于解释。

### 7.4 输出与排序规则

- NSGA-II 选择：
  - `paretoRank`：非支配层级（1 最优）
  - `crowding`：拥挤距离（越大越保多样性）
- 最终排序 key：
  1) `totalScore` 降序
  2) `paretoRank` 升序
  3) `crowding` 降序
  4) `signature` 升序（稳定）

输出 `table.rows[]` 典型字段：
- `rank, signature, source, qualified, N, B, wb, ws, coveragePct`
- `tonnageTotal, effScore, recScore`
- `effNorm, recNorm, distNorm, distPoints`
- `totalScore, paretoRank, crowding`

## 8. 缓存与一致性策略

### 8.1 前端 cacheKey：口径版本化（避免混用）

- efficiency：`buildEfficiencyCacheKey`
  - 包含：axis、边界 hash、wb 取最小、ws 范围、wsEnum=coarse9、B 范围、searchProfile、评分版本 v2、排序 spec 等。
- disturbance：`buildDisturbanceCacheKey`
  - 包含：axis、边界 hash、wb 取最大、ws 范围、B 范围、distSeq spec（序列枚举版本）等。
- recovery：`buildRecoveryCacheKey`
  - 包含：axis、边界 hash、wb/ws 取最小、厚度口径（grid/constant/none + data hash）、per-face/cleanup/fullCover 参数等。

> 这些 key 的作用是“口径冻结”：同 key 必须产生同结果；口径升级必须改 key。

### 8.2 后端缓存

- Node harness 缓存：按 payload 的 stable hash 缓存（避免重复跑 node）。
- weighted 缓存：按 weighted `cacheKey` 缓存整包结果。
- tonnage 缓存：调用方可通过 `cacheKey` 做上层缓存；后端自身主要做计算并返回。

## 9. 常见失败原因与排查建议

- `NO_CANDIDATES`（weighted）：
  - efficiency/recovery/disturbance 任一子任务返回空或全部不合格。
  - 前端会把 `debug.subResults` 摘要拼到 message 里，优先看：
    - 边界是否有效
    - axis 是否与 per-mode cacheKey 对齐
    - faceWidthMin/Max、wb/ws 范围是否为空/NaN（前端已做兜底，但快照导入仍需留意）
- `NO_ODI_FIELD`（disturbance-only 或 weighted 需要扰动）：
  - ODI 场未生成或 pack 不完整（field/gridW/gridH 缺失）。
- 吨位为 0 / `OMEGA_MISSING`：
  - 候选 render 缺少 omega/faces loops，或几何裁剪后为空。
  - thickness 无有效 cell（grid 全 NaN 或 <=0）且 constantM 未提供。

## 10. 版本与口径变更提示

- efficiency 评分已固化为 v2（cacheKey 中包含 `effScore=v2(...)`）。
- smart-resource 厚度场协议冻结为 v1.0（field[j][i] 的方向/翻转规则不可随意改动）。
- recoveryScore 在后端做了展示校准与 fullCoverMin 的温和惩罚（属于产品化口径的一部分）。
